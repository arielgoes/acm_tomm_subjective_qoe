# ACM TOMM Subjective QoE Evaluation — Design

**Date:** 2026-06-19
**Status:** Approved (brainstorming)

## Goal

A subjective video-quality study where participants compare **real** CGReplay
gameplay clips against **synthetic** (RIFE frame-interpolated) versions of the
same clip. Hosted as a static site on GitHub Pages, mirroring the look and flow
of `cgreplay_demo`, but simpler.

Each participant:
1. Lands on a page that generates a random participant ID.
2. Evaluates **all 15 pairs** (3 games × 5 bandwidths), in an order shuffled
   per-participant.
3. For each pair, rates each video 1–5 (Mean Opinion Score) and answers **one**
   question: *"Which videos do you think are real?"* → `None / Both / Video A /
   Video B`.

## Scope decisions (from brainstorming)

- **Storage:** Supabase (free Postgres + auto REST API) for online collection,
  with a localStorage + CSV-download fallback for local testing. Both sit behind
  a single `saveResponse()` function — the only code that knows about storage.
- **Pairs per user:** all 15, order shuffled per-user (seeded by participant ID).
- **Side (A/B):** which side shows the real vs synth video is randomized per
  pair, seeded by participant ID, so a later participant cannot memorize "A is
  always real." Recorded explicitly.
- **Pair rule:** for each `(game, bandwidth)`,
  real = `{bw}Mbit_Loss0_{game}.mp4`,
  synth = `{bw}Mbit_Loss0_interpolated_frames_rife_1600_900_{game}.mp4`.
  `Loss1` videos were removed and are never used. `reference_*` videos are
  ignored.
- **Manifest:** `pairs.json` is auto-generated by `generate_pairs.py` scanning
  `videos/`. A pair appears only if **both** its files exist (so today: 3 pairs
  at 4Mbit; the other 12 appear when 2/6/8/10 Mbit files are added).
- **No free-text comments** (the study asks one question only).
- **No statistics dashboard** in this build — a separate follow-up.

## Architecture

```
acm_tomm_subjective_qoe/
├── index.html              # landing + evaluation flow (vanilla JS, single file)
├── pairs.json              # AUTO-GENERATED manifest of available pairs
├── generate_pairs.py       # scans videos/, writes pairs.json
├── app.py                  # tiny local static server for testing
├── supabase_schema.sql     # table + insert-only RLS policy (paste once)
├── export_responses.py     # pull all rows from Supabase -> CSV
├── tests/                  # pytest unit tests for the Python pieces
├── videos/                 # the .mp4 files (Loss0 real + synth)
├── README.md               # setup: Supabase keys, generate, deploy
└── .github/workflows/      # (a) Pages deploy  (b) regenerate pairs.json
```

### Data flow

participant lands → random ID (persisted to localStorage) → `index.html` fetches
`pairs.json` → seeds a PRNG from the ID to (a) shuffle pair order and (b) pick
left/right per pair → for each pair: MOS-A, MOS-B, "which are real?" → on submit,
`saveResponse()` writes **one row per pair** to Supabase (localStorage mirror +
CSV-download fallback for local mode) → after pair 15, thank-you screen shows the
ID.

### Key boundary — `saveResponse(row)`

The only place that knows about storage:
- If `window.SUPABASE_URL` / `SUPABASE_ANON_KEY` are set → `POST /rest/v1/responses_acm_tomm_subjective_qoe`
  with `Prefer: return=minimal`; on failure, mirror to localStorage and surface a
  retry — never lose a completed pair.
- If not set (local testing) → append to a localStorage array; thank-you screen
  offers **Download CSV**.

## Manifest format (`pairs.json`)

```jsonc
{
  "version": "1.0.<unix_ts>",
  "generated_at": "2026-06-19T...Z",
  "hash": "<short hash of file list>",
  "pairs": [
    {
      "id": "4Mbit_Forza",
      "game": "Forza", "bandwidth_mbit": 4,
      "real":  "videos/4Mbit_Loss0_Forza.mp4",
      "synth": "videos/4Mbit_Loss0_interpolated_frames_rife_1600_900_Forza.mp4"
    }
  ]
}
```

- Real regex: `^(\d+)Mbit_Loss0_(Forza|Fortnite|Kombat)\.mp4$`
- Synth regex: `^(\d+)Mbit_Loss0_interpolated_frames_rife_1600_900_(Forza|Fortnite|Kombat)\.mp4$`
- A pair is emitted only when both real and synth exist.

## Deterministic randomization

- `seed = hashStringToSeed(participantId)` (djb2), feeds a `mulberry32` PRNG.
- **Order:** seeded Fisher–Yates shuffle of the pairs array.
- **Side:** per pair, a seeded coin flip decides whether `real` or `synth` is
  Video A (left). Recorded as `video_a_kind` / `video_b_kind`.

## Data model — one row per (participant, pair)

| Column | Type | Notes |
|---|---|---|
| `id` | bigint identity | PK |
| `created_at` | timestamptz | default `now()` |
| `participant_id` | text | random landing-page ID |
| `pair_id` | text | e.g. `4Mbit_Forza` |
| `game` | text | Forza / Fortnite / Kombat |
| `bandwidth_mbit` | int | 2/4/6/8/10 |
| `pair_index` | int | position in this user's shuffled order (0–14) |
| `video_a_kind` | text | `real` or `synth` shown as A |
| `video_b_kind` | text | `real` or `synth` shown as B |
| `video_a_file` | text | filename shown as A |
| `video_b_file` | text | filename shown as B |
| `score_a` | int | MOS 1–5 |
| `score_b` | int | MOS 1–5 |
| `which_real` | text | `None` / `Both` / `Video A` / `Video B` |
| `is_correct` | bool | derived client-side |
| `pairs_version` | text | from `pairs.json` |
| `pairs_hash` | text | from `pairs.json` |
| `user_agent` | text | optional QA |

RLS: enable, single policy — `INSERT` for `anon` with `WITH CHECK (true)`; no
select/update/delete (append-only).

## Screens

1. **Landing** — title, generated ID (persisted), Start button. No dashboard.
2. **Evaluation** — "Pair n of 15", Video A / Video B with controls + loop,
   1–5 star widget under each, single question
   `None · Both · Video A · Video B`. Next disabled until both scores + question
   answered; saves the row on click; last pair reads **Finish**.
3. **Thank-you** — confirms, re-shows ID; in local mode shows **Download CSV**.

## Error handling

- Empty/missing `pairs.json` → friendly "No video pairs available yet."
- Video 404 → inline error on that `<video>`.
- Supabase POST failure → mirror row to localStorage, show "saved locally, will
  retry" + manual retry.
- Refresh mid-session → same ID from localStorage ⇒ same shuffle; completed
  `pair_id`s stored to resume where left off.

## Testing

- `generate_pairs.py` — pairing correctness, ignores `reference_*`/unknown,
  includes a pair only when both files exist, stable hash.
- Seeding — deterministic order + side from an ID; `is_correct` correct for all
  four answers across both side assignments. (JS logic mirrored in a small
  Python reference test where practical; primary coverage is manual + the
  generator/export tests.)
- `export_responses.py` — CSV shaping from a sample API payload.
- Manual: run `app.py`, complete a session in local-CSV mode, verify the CSV;
  then a Supabase smoke test.
